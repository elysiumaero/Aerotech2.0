## Feature 4 — AES-128-CBC Encrypted Connection

> This test requires reflashing the ESP32 with ENCRYPT_ENABLED 1.
> Keep encryption DISABLED for all other tests and only enable here.

### 4A — Generate and set AES key

```bash
# Generate a random 16-byte key (32 hex chars):
python3 -c "import os; print(os.urandom(16).hex())"
# Example output: a3f1b2c4d5e6f7081920a1b2c3d4e5f6
```

```
[ ] Copy the generated key (exactly 32 hex characters)

[ ] Edit SUDARSHAN-RADHA/GCS/credentials.py:
        AES_KEY = "a3f1b2c4d5e6f7081920a1b2c3d4e5f6"   ← your key
        ENCRYPT_ENABLED = True

[ ] Edit credentials.h:
        #define AES_KEY_HEX  "a3f1b2c4d5e6f7081920a1b2c3d4e5f6"  ← same key
        #define ENCRYPT_ENABLED 1

[ ] Confirm keys are IDENTICAL character-for-character ✓
[ ] Neither file is committed (git status shows them as untracked/ignored) ✓
```

### 4B — Flash and verify

```
[ ] In Arduino IDE: open ATLAS_ESP32_Bridge_v2.ino
[ ] Compile and flash to ESP32
[ ] Open Serial Monitor (115200 baud)
    Expected line: [CRYPTO] AES-128-CBC ENABLED  key_len=16

[ ] Launch GCS:
        python SUDARSHAN-RADHA/GCS/radha_gcs.py

[ ] GCS connects to ESP32 → status shows CONNECTED ✓
[ ] GCS log shows normal telemetry (decrypted transparently) ✓
[ ] Capture raw TCP traffic (Wireshark or netcat):
        nc 192.168.4.1 5760
    → Data should appear as base64 blobs, NOT readable JSON ✓
```

### 4C — Key mismatch (negative test)

```
[ ] Change only GCS credentials.py to a different key
[ ] Relaunch GCS and connect
    [ ] GCS log shows JSON parse errors or garbled output ✓
    [ ] Telemetry NOT displayed (decryption fails silently, line skipped) ✓
[ ] Restore correct key → reconnect → works again ✓
```

### 4D — Fallback to plaintext

```
[ ] Set ENCRYPT_ENABLED = False in credentials.py
[ ] Set #define ENCRYPT_ENABLED 0 in credentials.h and reflash ESP32
[ ] Reconnect GCS → works normally without encryption ✓
[ ] Run all unit tests again — all 29 pass ✓
```

> **Leave ENCRYPT_ENABLED = 0 / False after this test section**
> so the rest of the bench tests work without reflashing.

## Feature 5 — GPS Source: Browser (no app) and GPS Server app

Two GPS input methods are supported. **Method A (Browser GPS) is the primary method**
and requires no app installation. Method B (GPS Server app) is a fallback.
Both feed the same GCS GPS panel and FC GPS pipeline.

---

### Method A — Browser GPS (no app required)

> The ESP32 hosts a web page at `http://192.168.4.1/`. The phone browser streams
> GPS data directly — no app installation needed.
>
> **Browser compatibility:**
> - iOS Safari ✓ (no setup)
> - Android Firefox ✓ (no setup)
> - Android Samsung Internet ✓ (no setup)
> - Android Chrome: one-time flag required (see 5A-3 below)

#### 5A-1 — Open the GPS page

```
[ ] On phone: Settings → WiFi → connect to SUDARSHAN_AP (password: radha2026)
[ ] Open phone browser (Safari / Firefox / Samsung Internet / Chrome)
[ ] Navigate to: http://192.168.4.1/
    [ ] Page loads — dark background, "SUDARSHAN GPS LINK" title ✓
    [ ] Shows LAT: —  LON: —  ACC: —  HDG: — ✓
    [ ] "GCS: checking..." text at bottom ✓
    [ ] "► START GPS" button visible ✓
```

#### 5A-2 — Stream GPS

```
[ ] Tap "► START GPS"
    [ ] Browser prompts for location permission → tap Allow
    [ ] Status changes to: "● GPS STREAMING" (green) ✓
    [ ] LAT and LON fill in with real values (not 0.000000) ✓
    [ ] ACC shows accuracy in metres (e.g. "ACC: 18m") ✓
    [ ] HDG shows heading or "N/A" if not moving ✓

[ ] ESP32 Serial Monitor shows:
        [HTTP ] GPS  lat=XX.XXXXXX  lon=XX.XXXXXX  acc=18m  fix=1  ✓

    [ ] LAT: XX.XXXXXX  (not 0.000000) ✓
```
